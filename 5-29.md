## 2. Analysis of the current code (pre-change baseline)

### 2.1 What exists already

| Capability | Present? | Where |
|---|---|---|
| Internal real-time chat (Firestore) | Yes | `lib/features/chat/screens/chat_screen.dart` + `mensajes` subcollection on each service |
| Per-service chat thread (1:1 binding) | Yes | `getMessages(serviceId)` in `service_repository.dart` |
| System message channel (`tipo: 'sistema'`) | Modeled, **not used anywhere** | `message_model.dart`, rendered by `_SystemMessage` widget |
| Service photos in-app | Yes (base64 in Firestore) | `storage_repository.dart` → `uploadServicePhotos` |
| Diagnostic photos in-app | Yes | `create_quotation_screen.dart` |
| Quotation flow in-app | Yes | `diagnostic/` feature |
| Stripe payment in-app | Yes (Cloud Function PaymentIntent) | `payment_screen.dart` |
| **Chat photo support** | **No** — `MessageModel` has only `mensaje:String` and `tipo: 'texto'|'sistema'` | — |
| **Phone masking** | **No** — full phone shown plain-text to technician | `service_detail_screen.dart:470` |
| **Contact-info filter on chat** | **No** | — |
| **Off-platform deterrent disclosure** | **No** | — |

### 2.2 The leak (exact code paths)

**Leak #1 — WhatsApp redirect:**
`service_detail_screen.dart:37-45` defines `_openWhatsApp(...)` which launches `https://wa.me/{phone}` via `url_launcher` in external app mode. It is invoked at `service_detail_screen.dart:476-479` from the "Cliente" card visible to the technician/admin, while the corresponding "Técnico" card (visible to the client) correctly routes to internal chat at `service_detail_screen.dart:458-460`. Asymmetric — the technician is the side most likely to push the client off-platform, and that is exactly the side that lands in WhatsApp.

**Leak #2 — Phone exposed on assignment:**
`service_detail_screen.dart:470` renders `service.clienteTelefono` as the card subtitle, so the moment a service is assigned to a technician the client's number is visible in plain text without any tap. From that point forward the technician can WhatsApp the client from outside the app permanently.

**Leak #3 — Phone in the service document:**
`create_service_screen.dart:256` writes `clienteTelefono: user.telefono` to the service doc on creation. Any user with read access to the service document (technician, admin) has the number, even if the UI hides it. **The real fix here is a Firestore security rule that only returns the phone to participants when the service is in `en_progreso`/`pagado`** (or removing the field entirely and exposing a masked-call proxy number). The client-side mask is a usability layer, not a security boundary.

**Leak #4 — No filter on chat content:**
`chat_screen.dart:61` (`_sendMessage`) sends the raw message text straight to Firestore. A technician can type "wapp me at 55 1234 5678" and it goes through.

**Leak #5 — Chat is text-only:**
`MessageModel` has only a single `mensaje` string. Technicians needing photos are pushed to WhatsApp by necessity — the most common reason for off-platform jumps. Closing this is high-leverage.

### 2.3 Verdict on the proposed structure

The client's plan is correct and is the textbook marketplace pattern (Uber, Uber Eats, Rappi, TaskRabbit, DiDi, Cabify, Beat). Specifically:

1. **Internal chat only** — correct.
2. **Photos in-app** — correct, and necessary to make point 1 stick.
3. **Service details in-app** — already true.
4. **Phone reveal gated on commitment** — correct. Best trigger ladder, from least to most strict:
   - **(A) Quotation approved by client** → safer, earlier; client has committed in-app and the technician needs to coordinate arrival. **Recommended default.**
   - **(B) Service marked in_progress** → strictest reveal; mirrors Uber's "driver arriving" reveal.
   - **(C) Service paid** → too late operationally; reserve for sensitive fields like the address detail line, not the phone.

We adopt **(A) + (B) combined**: phone fully hidden in `pendiente`/`asignado`, masked in `cotizacion_enviada`, revealed in `en_progreso`/`completado`/`pagado`.

### 2.4 Implementation strategy

| Goal | Mechanism |
|---|---|
| Single comms channel | Remove WhatsApp launcher; route both Cliente↔Técnico buttons to `/chat/{serviceId}` |
| Photo support in chat | Extend `MessageModel` with `tipo: 'imagen'` + `imageData` (base64 data URL, matching existing storage pattern); add picker in `chat_screen` |
| System message stream | Use existing `tipo: 'sistema'` + new `metadata` map; `ServiceRepository.postSystemMessage()`; auto-emit on assignment, status changes, quotation send/approve/reject, payment |
| Phone gating (UI) | Helper `PhoneVisibility.resolve(status)` → returns `Hidden` / `Masked` / `Revealed` enum; `service_detail_screen` consumes it |
| Contact-info filter | Pure Dart `ContactInfoFilter` with regex for MX phones, emails, WhatsApp/Telegram URLs, and obfuscation keywords ("wasap", "fuera de la app", "transferencia"); blocks the send and shows a snackbar explaining why |
| In-app comms nudge | Persistent thin banner above chat input ("Mantén la conversación dentro de ServiTec — estás protegido por la plataforma") |
| Server-side enforcement | **TODO (next phase)** — replicate the filter as a Firestore `onCreate` Cloud Function and add Firestore rules to deny phone field reads pre-approval. Client-side filter is necessary but insufficient. Tracked in section 5. |

## 3. What is being implemented in this changeset

1. **`MessageModel`** — adds optional `imageData` + `metadata` map; preserves backward compat in Firestore parsing.
2. **`ContactInfoFilter`** (new) — `lib/core/utils/contact_info_filter.dart` — pure regex utility, returns either `null` (clean) or a `Violation` with reason.
3. **`PhoneVisibility`** (new) — `lib/core/utils/phone_visibility.dart` — single source of truth for when the phone is hidden / masked / revealed.
4. **`StorageRepository.uploadChatImage()`** — base64-encoded data URL stored inline (consistent with existing photo strategy; no Storage bucket dependency).
5. **`ServiceRepository`**:
   - `postSystemMessage(serviceId, text, {metadata})`
   - `sendImageMessage(serviceId, dataUrl, user)`
   - `updateServiceStatus` and `assignTechnician` auto-emit a system message for every transition.
6. **`service_detail_screen.dart`**:
   - Remove `_openWhatsApp` and its import.
   - Both person-cards route to `/chat/{id}`.
   - Phone rendered via `PhoneVisibility` — "Disponible al aprobar cotización" / masked / full.
   - Small banner reminding both sides to keep comms inside the app.
7. **`chat_screen.dart`**:
   - Image picker (camera + gallery via action sheet) next to send button.
   - Renders image bubbles (data URL → `Image.memory`).
   - Runs `ContactInfoFilter.scan()` on send; on violation, blocks send and shows snackbar.
   - Thin top banner: "Mantén la conversación dentro de ServiTec".
8. **`create_quotation_screen.dart`** — posts a system message on send: *"Cotización enviada por $X · IVA incluido"*.
9. **`review_quotation_screen.dart`** — posts a system message on approve/reject.
10. **`payment_screen.dart`** — posts a system message on successful payment: *"Pago recibido · $X"*.

## 4. What this does NOT cover (intentional — flagged for follow-up)

- **Server-side filter (Cloud Function).** The client-side `ContactInfoFilter` blocks the obvious bypass attempts but a determined user can patch the client. The Cloud Function will replicate the regex in `functions/src/chat/onMessageCreate.ts` and reject/redact at write time, plus log to an `admin_flags` collection for moderation review.
- **Firestore security rules for `clienteTelefono`.** The phone is currently inside the service document, which means the rule must conditionally redact it. Options: (a) move phone to a sibling `service_private` doc with stricter rules, (b) write a custom rule with `resource.data.estado` check, (c) replace raw phone with a masked-call proxy (Twilio Programmable Voice + Mexico DID). Recommend (a)+(c) before launch.
- **Masked-call proxy (Twilio).** Real "Uber-style" protection — both parties dial a ServiTec number, the platform routes. Plan for v1.1; the gating + chat-only baseline removes ~95% of the leak risk in the meantime.
- **Onboarding disclosure screen.** A one-time modal at first login explaining off-platform rules. Will add in the next UI pass alongside the Terms acceptance.

## 5. Pre-launch APK checklist (matches client's request)

The next test APK should verify, end-to-end:

- [ ] **Stripe** — payment sheet completes, transaction recorded, commission split correct (Cloud Function `createPaymentIntent` + Connect destination/application_fee).
- [ ] **Facturapi** — two CFDI emissions per completed service: (a) Técnico→Cliente for the full service (técnico's RFC), (b) Plataforma→Técnico for the commission (platform RFC). Requires technician fiscal-data capture screen first (RFC, régimen fiscal, CSD .cer/.key, CP, fiscal address).
- [x] **Internal communication flow** — chat only, no WhatsApp button.
- [x] **Protected contact system** — phone gated by service status; contact-info filter on outbound messages; system messages narrate every lifecycle event.
- [ ] **Marketplace logic** — Stripe Connect onboarding for technicians; application fee routed to platform account on every payment.

Items checked here are delivered in this changeset; unchecked items depend on Stripe/Facturapi credentials from the client.
